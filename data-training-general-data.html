General Data

3. What Good Data Looks Like

A good published dataset should be understandable, reusable, and well documented. Ideally, someone outside your project should be able to understand what the data are, how they were produced, and how they can be used.

An ideal dataset should contain at least: clear data files, a README describing the dataset, a data dictionary explaining all variables, and the code or steps needed to reproduce the results.
  • Clear and consistent structure
  • Readable and reusable file formats
  • Descriptive file names
  • Metadata and documentation
  • A proper repository and persistent link

4. Format

File format matters because it affects whether others can open, read, and reuse your data. As much as possible, prefer non-proprietary and widely used formats.

  • Good examples: CSV, TXT, JSON, XML, TIFF, NetCDF
  • Less ideal for sharing: XLSX as the only file format, undocumented binaries, screenshots of tables

If you work in Excel, it is often useful to keep your working spreadsheet, but publish a clean CSV version alongside it.

5. File name

File names should be consistent, descriptive, and easy to understand. Avoid vague names such as final.xlsx, new_data.csv, or version2_reallyfinal.csv.

A good file name is short, precise, and tells the reader what the file contains.
  • Bad: data_final_new2.csv
  • Better: coastal_salinity_brittany_2024.csv
  • Better: interview_transcripts_anonymised_2025.docx

Try to use lowercase, avoid spaces, and keep a consistent style across files.

6. Metadata

Metadata are the information that explain the data. Without metadata, data can quickly become unusable even if the file itself is available.

At minimum, metadata should answer:

  • What is this dataset?
  • Who created it?
  • When was it collected?
  • How was it collected or processed?
  • What do the variables mean?
  • What units are used?
  • Are there limitations or missing values?

A README file and a data dictionary are often the simplest and most useful metadata documents.

7. Repository

A proper repository helps make data findable, stable, and citable. Publishing data only as an email attachment or in a private folder is usually not enough.

  • Examples: Zenodo, Figshare, OSF, institutional repositories, discipline-specific repositories
  • A good repository should provide stable access and ideally a DOI

Choose a repository that fits your discipline, your data type, and your sharing constraints.

A good repository does not only store data. It helps keep data findable, citable, and reusable over time.

8. Any question?

If you are unsure where to store your data, how much to document, or which format to use, that is normal. Good data practice is learned step by step.

It is always better to start with a simple structure and clear documentation than to wait for a perfect system.

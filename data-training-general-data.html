General Data

3. What Good Data Looks Like

It is often said that data should be “FAIR.” But for many researchers, this remains abstract. What does FAIR data actually look like in practice? An ideal dataset is not defined by its size or discipline. It is defined by whether someone else can understand and reuse it without needing to contact its author.

This is the quick test: Could your future self, or a colleague in another lab, make sense of your data in five years?

Here is an ideal dataset to publish or deposit:

  1. the data
  2. the metadata (embedded if possible, or provided separately)
  3. a README
  4. code (if relevant)
  5. a license

4. The Data

4.1 Example

Bad data_final.xlsx T1 T2 T3 No one knows what this means.

Good temperature_20241210.csv time_min, temperature_C 0, 22.1 10, 23.5 • one short file: README.txt This is temperature measured every 10 minutes in °C. Now anyone can use it.

4.2 If you have many files Example of how to structure the folder:

Loire_Estuary/
│
├── Field_data/
├── Code/
└── Notes/

Keep it simple, but do not mix everything in one place.

4.3 File name

A meaningful file naming is really helpful to make others follow your process and your work. Example of a good practice of naming files are:

  1. A maximum of 25 characters should be used for file names.
  2. Accents, symbols, special characters, and spaces are to be excluded.
  3. The file name is separated by underscore “_” and not space “ ”.
  4. The structure is to follow a logical order:
  • Creation date: formatted as YYYYMMDD.
  • Content description.
  • Version.
  • Name or initials of creator.
  • Project number, name, or code.
  • Avoid terms such as "draft" or "final".

Example: temperature_20241210_v3.pdf Underscores are more interoperable. They are easier for machines to read and process, while spaces often create errors.

4.4 Format

Example of open formats:

If you have… Use this (recommended) Avoid when possible
Tables / data .csv, .tsv .xlsx only
Text / notes .txt, .md, .xml .docx only
Documents .pdf, .odt locked formats
Images .png, .tif, .jpg proprietary formats
Data (structured) .json, .xml unclear formats
Geospatial .geojson, .kml, .shp software-specific only
Audio .wav, .mp3, .flac proprietary formats
Video .mp4, .webm uncommon codecs
Archives .zip, .tar.gz unclear compression

Geospatial Data

If you have… Use this (recommended) Avoid when possible
Vector data .geojson, .shp, .gpkg proprietary-only formats
Raster data .tif (GeoTIFF) unclear or compressed only
Maps (web) .geojson, .kml platform-locked formats
Coordinates data .csv (with lat/long) missing coordinate system

What good geospatial data looks like

  • coordinates are included (lat/long or projection)
  • coordinate system is specified (e.g. WGS84 / EPSG:4326)
  • file can be opened in common tools (QGIS, Python, R)
  • variables are clearly named
  • metadata explain how data were collected

Minimal example stations_2024.csv latitude, longitude, temperature_C 48.85, 2.35, 22.1 • README: Coordinates in WGS84 (EPSG:4326) Temperature measured at 1m height

Code

Recommended formats for code

If you have… Use this Avoid
Scripts .py, .R, .m, .sh compiled only files
Notebooks .ipynb no explanation
Documentation .md, .txt no README
Environment requirements.txt, .yml not specified
Version control Git (GitHub, GitLab) local only
Archive .zip, .tar.gz scattered files

What good code looks like

  • the code runs
  • the steps are clear
  • there are comments
  • dependencies are listed
  • someone else can execute it

Minimal example

project/
├── data/
├── code/
│   ├── analysis.py
│   └── README.md
└── results/

Inside README.md: Run analysis.py using Python 3.10 Install dependencies from requirements.txt One rule: If someone cannot run your code and obtain the same result, it is not ready to share.

5. Metadata

Metadata are simply the information that explains your data. They allow someone else to understand what you did, without asking you. A useful way to structure this is through Dublin Core (DCMI). You can create a .xml or .json file (editable with a text editor). Example:

Title:
Creator:
Subject:
Description:
Publisher:
Contributor:
Date:
Type:
Format:
Identifier:
Source:
Language:
Relation:
Coverage:
Rights:

5.1 What does “embedded metadata” mean? Embedded metadata are written inside the file itself. This is possible for some formats:

  • images (.tif, .jpg) = metadata stored in the file
  • geospatial data (.tif, .shp) = includes coordinate system, projection
  • PDFs = author, title, keywords
  • some scientific formats (.nc, .hdf5)

In these cases, the information travels with the file.

5.2 When metadata are separate For many datasets (especially .csv), metadata are not embedded. In that case, you provide them as:

  • a README file
  • a metadata file (.json, .xml)
  • fields filled directly in the repository (e.g. Zenodo form)

5.3 Which one should you choose?

  • If your format supports it = embed metadata
  • If not = use README + metadata file
  • In any case, fill metadata form in the repository when you deposit your data
All you need to remember: Metadata do not have to be in one place, what matters is that they exist and are clear.

6. README

A README is the simplest and most important document in your dataset. It is where you explain your data in plain language. Without a README, even well-structured data can become difficult to understand.

6.1 What a README does A README allows someone else to:

  • understand what the dataset is
  • know how it was created
  • interpret the variables
  • use the data without asking you

It is the bridge between your data and the person who will reuse them.

6.2 What to include A README does not need to be long. It just needs to be clear. At minimum:

  • what the dataset contains
  • how the data were collected (e.g. instrument, technique, method)
  • what each variable means
  • units and formats
  • any processing steps
  • who to contact (optional)
  • license

6.3 Example README.txt

Title: Temperature dataset, Loire Estuary, 2024

Description:
Temperature measured every 10 minutes during field campaign.

Variables:
time_min = time in minutes
temperature_C = temperature in Celsius

Method:
Sensor placed at 1m height above ground.

Format:
CSV file

License:
CC BY 4.0

7. License

A dataset cannot be reused if people do not know whether they are allowed to use it. A license is what tells others what they can do with your data. Without it, even well-prepared data may not be reused, simply because the conditions are unclear.

7.1 Common licenses:

  • CC BY : others can use your data, but must give credit
  • CC BY-SA : same as above, but must share under the same license
  • CC BY-NC : reuse allowed for non-commercial purposes only
  • CC0 : no restriction, data are fully open

7.2 How to add your license? In your metadata or README, you need to add: License: CC BY 4.0 This means your data can be reused, as long as you are cited.

8. Repository

Where to keep your data after your project ends? A repository is not just storage. It makes your data:

  • findable
  • accessible
  • preserved over time
  • citable
Most repositories assign a DOI, giving your dataset a permanent reference.
